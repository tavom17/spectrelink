# API Gateway & Database Architecture
## Session Notes — 2026-03-26

This document captures the architectural decisions, clarifications, and mental models developed during the design session covering the api-gateway and database layer for Spectrelink.

---

## 1. The Request Flow

A request travels: `frontend → nginx → api-gateway → [service] → api-gateway → Postgres/Redis`

Key principle: **services never touch the database directly.** Only the `api-gateway` sits on the `data` network. Services (wallet-generator, coin-launcher, liquidity-app) handle business logic only — they receive instructions and return results. The gateway owns all persistence.

---

## 2. Custody Model

**Spectrelink is a custodial platform.**

- The platform generates and holds all wallet seeds, encrypted.
- Users deposit SOL into platform-managed wallets.
- The platform executes all on-chain transactions on behalf of users.
- Users can withdraw from custodial wallets back to their personal wallets at any time.

### Two Types of Wallets

| Type | Description | Who Controls Seed |
|------|-------------|-------------------|
| Platform Operational | Funder + slave wallets managed by Spectrelink for bundling/launching | Spectrelink (server-side, encrypted) |
| User Personal | The user's own external wallet (Phantom, Ledger, etc.) | User only — Spectrelink never sees it |

The custody model applies to **platform operational wallets**. The user's personal wallet is never touched by Spectrelink.

### Funder / Slave Architecture

- **Funder wallet** — index 0 on the derivation path. User deposits SOL here. Distributes to slave wallets for operations.
- **Slave wallets** — indices 1–N. Execute transactions (buys, sells, liquidity operations) independently.
- All derived from the same per-user HD seed using BIP-44 derivation paths (e.g., `m/44'/501'/0'/0'` for Solana).

---

## 3. HD Wallet Model

- Each user gets one BIP39 mnemonic seed, generated by the platform at registration.
- Wallets are derived deterministically from the seed using a derivation index.
- The seed is the master key — storing it means storing access to all current and future wallets for that user.
- The platform stores: `encrypted_seed`, `public_key` per wallet, `derivation_index` per wallet.
- **Private keys and mnemonics are never returned to the user or stored in plaintext.**

### Why Per-User Seed (Not Master Platform Seed)

Blast radius isolation: if one user's seed is compromised, only that user is affected. A single platform master seed breach would expose every user simultaneously.

### Derivation Index Assignment

When a user creates a new wallet, the gateway:
1. Queries `wallets` to find the highest existing derivation index for that user
2. Assigns `next_index = max_index + 1`
3. Passes the index to `wallet-generator` to derive the keypair
4. `wallet-generator` returns the public key
5. Gateway stores `public_key` + `derivation_index` in the `wallets` table

Derivation indices must be sequential and never reused per user.

## 4. Database Schema

Four tables with clear, single responsibilities.

### `users`
Identity and authentication credentials only.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID / serial | Primary key |
| username | varchar | Unique |
| password_hash | varchar | bcrypt or argon2 — never plaintext |
| created_at | timestamp | |
| is_active | boolean | For account suspension |

### `user_seeds`
Custody layer. 1:1 with users. Restricted access — only the privileged role used for seed operations should have SELECT on this table.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID / serial | Primary key |
| user_id | FK → users.id | Unique (1:1) |
| encrypted_seed | text | AES-encrypted at application layer before DB write |
| created_at | timestamp | |

### `wallets`
Derived wallet addresses. Many per user.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID / serial | Primary key |
| user_id | FK → users.id | |
| public_key | varchar | Solana base58 address, unique |
| derivation_index | integer | Sequential per user, never reused |
| wallet_type | enum | `funder` or `slave` |
| created_at | timestamp | |

### `refresh_tokens`
Session management. Enables JWT revocation.

| Column | Type | Notes |
|--------|------|-------|
| id | UUID / serial | Primary key |
| user_id | FK → users.id | |
| token_hash | varchar | Hash of the refresh token — never store raw token |
| expires_at | timestamp | |
| revoked_at | timestamp | Nullable — set on logout or compromise |
| created_at | timestamp | |

## 5. Encryption & Security Model

### Password Storage
Passwords are hashed with **bcrypt** (or argon2) before being written to the database. The hash is irreversible — at login, the incoming password is hashed and compared. The original password is never stored or recoverable.

### Seed Encryption
Seeds are encrypted at the **application layer** before reaching Postgres. Postgres only ever stores ciphertext.

- Encryption key lives in the `api-gateway`'s environment variables (`.env`)
- The key must NEVER be stored in the same database as the ciphertext it protects
- Principle: **separation of key and ciphertext** — a Postgres dump without the application key is useless

### Postgres Role Permissions
- Application DB user: `SELECT/INSERT/UPDATE` on `users`, `wallets`, `refresh_tokens`
- Privileged seed role: additionally has `SELECT/INSERT` on `user_seeds`
- Restricting `user_seeds` at the DB role level means a compromised query path cannot access seeds even if it bypasses application logic

## 6. Authentication & Session Model

### Registration Flow
1. User submits username + password
2. Gateway hashes password (bcrypt)
3. `wallet-generator` generates a BIP39 mnemonic
4. Gateway encrypts the mnemonic with the `.env` key
5. **Database transaction** — atomically writes to `users` AND `user_seeds`
   - If either write fails, both roll back (ACID Atomicity)
6. Wallet creation is **separate** — user creates wallets later from the portal

### Login Flow
1. User submits username + password
2. Gateway fetches `password_hash` from `users`
3. bcrypt compare — valid or reject
4. On success: generate access token (JWT) + refresh token
5. Store refresh token hash in `refresh_tokens` table
6. Return: access token in response body, refresh token as **httpOnly cookie**

### JWT Token Strategy
Two tokens, two jobs:

| Token | Expiry | Storage | Revocable |
|-------|--------|---------|-----------|
| Access token | 15 minutes | JavaScript memory only (not persisted) | No — expires naturally |
| Refresh token | 7–30 days | httpOnly cookie + hash in DB | Yes — set `revoked_at` in DB |

- **Why httpOnly cookie:** JavaScript cannot read httpOnly cookies. Immune to XSS token theft. Browser sends it automatically.
- **Why access token in memory:** Lost on page refresh, which is fine — the refresh token issues a new one. Never persisted to localStorage (XSS vulnerable).
- **Revocation:** To invalidate a session (logout, compromise response), set `revoked_at` on the refresh token row. The access token expires within 15 minutes; without a valid refresh token, no new access token can be issued.

### Redis Role in Auth
- **JWT blacklist on logout** — write invalidated token IDs to Redis. Auth middleware checks blacklist before accepting a token. Solves the "can't revoke a stateless JWT" problem.
- **Rate limiting state** — request counts per user per time window in Redis (no Postgres rows needed)
- **Auth check caching** — avoid hitting Postgres on every protected request
- **Wallet balance cache** — Solana RPC calls are expensive; cache with short TTL (e.g., 30s)

## 7. API Gateway Architecture

### Route Groups

| Prefix | Responsibility |
|--------|---------------|
| `/auth` | Registration, login, logout, token refresh |
| `/wallets` | Wallet creation, listing, balance queries |
| `/launcher` | Coin/token launch operations |
| `/liquidity` | Liquidity management operations |

### Authorization Middleware
Authorization is **not a route** — it is a middleware layer (Fastify `preHandler` hook) that runs before every protected route handler.

Middleware checks in order:
1. JWT present in request? → 401 if not
2. JWT signature valid and not expired? → 401 if not
3. Token in Redis blacklist? → 401 if yes
4. User account active? → 403 if not
5. User authorized for this specific action? → 403 if not
6. Pass to route handler

### Signing / Withdrawal Flow
The gateway is the only service that ever touches decrypted seed material:
1. User submits withdrawal request (destination address + amount)
2. Auth middleware validates identity and authorization
3. Gateway fetches encrypted seed from `user_seeds`
4. Decrypts in memory using `.env` key
5. Derives the relevant wallet keypair (via `wallet-generator`)
6. Constructs and signs the transaction
7. Broadcasts to Solana via Helius
8. Decrypted seed discarded from memory
9. **The seed never travels between services — decryption and signing authority stays in the gateway**

---

## 8. Monolithic Gateway — Deliberate Choice

Spectrelink starts with one `api-gateway`. The reasoning:

- Traffic patterns are unknown at this stage — premature splitting adds distributed systems overhead with no benefit
- Multiple gateways means multiple auth middleware implementations that can drift out of sync
- **Modular internals** (each route group in its own folder/module) means extraction into separate services is a refactor later, not a rewrite

A split into separate gateways (auth-service, wallet-gateway, etc.) is appropriate when a specific route group has independent scaling or deployment needs. Spectrelink does not have those needs yet.

## 9. Key Concepts Reference

| Concept | Definition |
|---------|-----------|
| **Custodial** | Platform holds and controls user wallet seeds |
| **Non-custodial** | User holds their own seed; platform never sees it |
| **HD Wallet (BIP-32/44)** | Hierarchical Deterministic wallet — entire keypair tree derived from one seed |
| **Derivation path** | Coordinate within the HD tree (e.g., `m/44'/501'/0'/0'`) |
| **Blast radius** | How many users/systems are affected if a given component is compromised |
| **ACID** | Atomicity, Consistency, Isolation, Durability — database transaction guarantees |
| **Application-level encryption** | Data encrypted before it reaches the DB; DB only stores ciphertext |
| **Separation of key and ciphertext** | Encryption key must never live in the same system as the data it protects |
| **httpOnly cookie** | Browser cookie inaccessible to JavaScript — immune to XSS token theft |
| **JWT stateless weakness** | Server holds no record of issued JWTs; stolen tokens cannot be directly revoked |
| **Refresh token pattern** | Short-lived access token (memory) + long-lived revocable refresh token (DB + cookie) |
| **Parameterized queries** | SQL queries where structure and data values are sent separately — prevents injection |
| **Middleware / preHandler** | Code that runs before route handlers; used for auth enforcement |
